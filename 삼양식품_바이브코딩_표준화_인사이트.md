## 바이브코딩이란?

### 정의
**AI와의 대화형 상호작용을 통해 비개발자도 코딩할 수 있게 하는 개발 방식**

### 핵심 특징
- 프로그래밍 지식 없이도 개발 작업 수행 가능
- PM, 디자이너도 코드 수정 및 기능 구현 가능
- AI를 "고급 알바"처럼 활용하는 개발 패러다임

### 전통적 개발 vs 바이브코딩

| 구분 | 전통적 개발 | 바이브코딩 |
|------|------------|-----------|
| **개발 주체** | 개발자만 코딩 | PM/디자이너도 코딩 |
| **소통 방식** | 기획 → 개발자 → 구현 | 기획 → AI → 즉시 구현 |
| **개발 속도** | 2주 소요 | 1일 완료 |
| **컨텍스트** | 개발자 머릿속 | 리포지토리 + AI |
| **의사결정** | 회의 → 문서화 → 개발 | 회의 → AI 스크립트화 → 즉시 개발 |

### 초기 팀 구성
```
👥 PM 1명 + 디자이너 1명 + 엔지니어 1명
```

- **5개월 만에** Popow.ai 미국 시장 정식 출시

### 성공 요인
> "개발자한테 설명할 시간에 커서한테 시키면 바로 작업이 끝나기 때문입니다."

## 맥도날드 시스템: 핵심 철학

### 🍔 맥도날드 프랜차이즈 모델 적용

삼양식품은 맥도날드의 프랜차이즈 시스템을 개발 프로세스에 도입했습니다.

```
맥도날드 본사 (메뉴얼 제공)
    ↓
지점장 (메뉴얼 실행, 품질 관리)
    ↓
알바 (정해진 업무 수행)
```

### 개발 조직 구조

| 맥도날드 | 바이브코딩 팀 | 역할 |
|----------|--------------|------|
| **본사** | 테크리드 | 매뉴얼(가이드) 제공 |
| **지점장** | 바이브코더(PM/디자이너) | 업무 관리 및 품질 검증 |
| **알바** | AI 에이전트 | 실제 코드 작성 |

### 핵심 원리

1. **표준화된 매뉴얼**: 200줄 내외의 핵심 가이드
2. **명확한 역할 분담**: 인간은 의도와 의사결정, AI는 실행
3. **휘발성 컨텍스트**: 리포지토리에 없는 정보는 버려도 됨

## 에이전트 개발 7단계 워크플로우

### 1️⃣ 컨텍스트 확보
```markdown
📌 목적: 프로젝트 현재 상태 파악

방법:
- 리포지토리 구조 분석
- 기존 코드베이스 이해
- 도메인 지식 습득
```

**실전 팁**:
- 회의 중 녹음기 켜두기
- 회의 종료 후 AI에게 스크립트 작성 요청
- "우리가 회의에서 이 결론을 도출한 과정을 스크립트로 작성해달라"

### 2️⃣ 요구사항 정의
```markdown
📌 목적: Must-have와 Nice-to-have 구분

구조:
- Goal: 이 기능으로 달성하고자 하는 것
- Non-Goal: 이번에 다루지 않을 범위
- 우선순위: Must / Should / Nice-to-have
```

**AI 활용**:
- 가상 FGI(Focus Group Interview) 시뮬레이션 실행
- 다양한 사용자 시나리오 검증
- 예상되는 엣지 케이스 발굴

### 3️⃣ UX 설계
```markdown
📌 목적: 사용자 경험 흐름 정의

포함 사항:
- 사용자 여정(User Journey)
- 인터랙션 시나리오
- 에러 처리 방안
```

**바이브코딩 방식**:
- 디자이너가 직접 AI와 대화하며 프로토타입 수정
- 즉각적인 피드백 루프

### 4️⃣ 의존성 분석
```markdown
📌 목적: 기술적 제약사항 파악

확인 사항:
- 기존 시스템과의 연결점
- 필요한 외부 API/라이브러리
- DB 스키마 변경 필요 여부
```

**토큰 절약 팁**:
```
도메인별로 쪼개서 관리:
- db-schema.md
- api-endpoints.md
- page-structure.md
```

### 5️⃣ 구현 계획
```markdown
📌 목적: 개발 순서 및 방법 결정

내용:
- 파일 구조 설계
- 컴포넌트/모듈 분리 전략
- 코딩 컨벤션 준수 방법
```

**핵심**: "기획서만 잘 쓰면, 뒷단은 정말 마법같이 잘 풀리는 경험을 할 수 있습니다."

### 6️⃣ 워크플로우 및 리스크 분석
```markdown
📌 목적: 예상 문제 사전 차단

분석 항목:
- 병목 지점 식별
- 보안 취약점 검토
- 성능 이슈 예측
```

**AI 에이전트 활용**:
- 다양한 실패 시나리오 시뮬레이션
- 리스크 우선순위 자동 분류

### 7️⃣ 테스트 작성
```markdown
📌 목적: 품질 보장 및 회귀 방지

테스트 레벨:
- 단위 테스트
- 통합 테스트
- E2E 시나리오 테스트
```

**자동화 전략**:
- PR 가이드에 테스트 체크리스트 포함
- AI가 자동으로 테스트 케이스 생성

## 실전 바이브코딩 스타일 가이드

### 🎯 인간(바이브코더)의 책임

#### 1. **의도 기록**
```markdown
❌ 나쁜 예:
"로그인 기능 추가"

✅ 좋은 예:
"사용자 이탈률이 높아 소셜 로그인을 추가합니다.
Google과 TikTok 로그인을 우선 지원하며,
이메일 로그인은 다음 스프린트로 미룹니다.
이유: 틱톡 크리에이터가 주 타겟이므로 TikTok OAuth가 핵심입니다."
```

#### 2. **결정 과정 명시**
```markdown
의사결정 템플릿:

## 결정: [선택한 옵션]

### 검토한 대안:
1. 옵션 A - [장점] / [단점]
2. 옵션 B - [장점] / [단점]
3. 옵션 C - [장점] / [단점]

### 선택 근거:
- [비즈니스 이유]
- [기술적 이유]
- [팀 리소스 고려]

### 트레이드오프:
- 포기한 것: [...]
- 얻은 것: [...]
```

#### 3. **우선순위 분류**
```markdown
Must-have (P0):
- [ ] 소셜 로그인 (Google, TikTok)
- [ ] 세션 관리

Should-have (P1):
- [ ] 비밀번호 재설정
- [ ] 이메일 인증

Nice-to-have (P2):
- [ ] 2FA 인증
- [ ] 생체 인증 지원
```

### 🤖 AI 에이전트의 책임

#### 1. **코드 작성**
- 가이드라인 준수
- 일관된 코딩 스타일
- 주석 최소화 (코드가 자기 설명적이어야 함)

#### 2. **테스트 자동 생성**
- 엣지 케이스 커버
- 회귀 테스트 추가

#### 3. **문서 업데이트**
- API 변경 시 자동 문서화
- 컴포넌트 Props 문서화

## 컨텍스트 관리 전략

### 🚨 직면했던 문제점

#### 문제 1: 환각 (Hallucination)
```
증상: AI가 존재하지 않는 함수나 API를 사용
원인: 컨텍스트 부족 또는 오래된 정보
```

**해결책**:
- 리포지토리 중심 개발: "모든 정보가 리포지토리 안에 있어야 합니다"
- 실시간 검증: AI에게 구현 상태 즉시 확인

#### 문제 2: 컨텍스트 과부하
```
증상: Claude.md 파일이 2,000줄 초과
영향: "컨텍스트의 30%가 이 룰 파일로 채워져 있는" 상황
```

**해결책**:
- 핵심 가이드 200줄 이내로 제한
- 어펜딕스로 세부 정보 분리

#### 문제 3: 의도 상실
```
증상: "누가 이 코드를 만들었는지, 도대체 무슨 의도로 만든 건지" 불명확
영향: 유지보수 어려움, 중복 작업
```

**해결책**:
- 모든 의사결정 PRD에 기록
- Git 커밋 메시지에 의도 명시

### ✅ 바이브코딩 환경의 3대 전제 조건

#### 1. **리포지토리 중심**
```
모든 정보는 리포지토리에 존재해야 합니다.

필수 파일:
- README.md (프로젝트 개요)
- ARCHITECTURE.md (시스템 구조)
- API_ENDPOINTS.md (API 목록)
- DB_SCHEMA.md (데이터베이스 구조)
- COMPONENTS.md (UI 컴포넌트 목록)
- CLAUDE.md (AI 작업 가이드 - 200줄 이내)
```

#### 2. **질의응답 시스템**
```
AI가 즉시 답변할 수 있어야 하는 질문들:

- "현재 어떤 API 엔드포인트가 있나요?"
- "사용자 인증은 어떻게 구현되어 있나요?"
- "이 컴포넌트의 의존성은 무엇인가요?"
- "최근 변경 사항은 무엇인가요?"
```

**구현 방법**:
- Cursor의 `@codebase` 기능 활용
- Claude Code의 컨텍스트 검색

#### 3. **빠른 배포**
```
통합 스택:
GitHub → Vercel (자동 배포)
       ↓
     Slack (알림)
       ↓
    Cursor (즉시 수정)
```

**목표**: 코드 푸시 → 프로덕션 반영까지 5분 이내

## 에이전트 운영 3대 원칙

### 원칙 1: 빠른 시작 (Quick Start)
```
"에이전트가 켜지면 200줄 내외의 핵심 가이드만 읽고 바로 일을 시작"
```

**CLAUDE.md 구조 예시**:
```markdown
# AI 에이전트 작업 가이드

## 프로젝트 개요 (20줄)
- 서비스 목적
- 타겟 유저
- 핵심 기능 3가지

## 코딩 컨벤션 (50줄)
- 파일명 규칙
- 컴포넌트 구조
- 네이밍 규칙
- Import 순서

## 필수 체크사항 (30줄)
- 보안 (XSS, CSRF 방지)
- 성능 (lazy loading)
- 접근성 (a11y)

## 자주 하는 작업 (50줄)
- 새 페이지 추가하기
- API 연동하기
- 상태 관리하기

## 어펜딕스 (50줄)
- 상세 문서 링크
- DB 스키마 위치
- API 문서 위치
```

### 원칙 2: 어펜딕스 참조 (Appendix)
```
핵심 가이드에 모든 걸 담지 않습니다.
필요할 때만 참조할 수 있도록 링크를 제공합니다.
```

**예시**:
```markdown
## API 연동하기

기본 패턴:
[코드 스니펫 30줄]

상세한 에러 핸들링은 → docs/api-error-handling.md 참조
인증 토큰 관리는 → docs/auth-flow.md 참조
```

### 원칙 3: 휘발성 컨텍스트 (Volatile Context)
```
"하나의 업무가 끝나면 컨텍스트를 즉시 잊어버리게"
"리포지토리에 남지 않는 정보는 휘발되어도 되는 정보"
```

**구현 방법**:
1. 에이전트 세션 종료 시 컨텍스트 리셋
2. 중요한 결정은 즉시 PRD나 코드 주석에 기록
3. 임시 메모는 삭제 (리포지토리를 어지럽히지 않음)

**장점**:
- 토큰 사용량 절감
- AI가 최신 정보만 참조
- 불필요한 컨텍스트 간섭 방지

## 토큰 최적화 기법

### 💰 토큰은 비용입니다

#### 전략 1: 도메인별 문서 분리
```
❌ 나쁜 예: 모든 정보를 하나의 파일에

docs/
  └── everything.md (5,000줄)

✅ 좋은 예: 도메인별로 쪼개기

docs/
  ├── db-schema.md (300줄)
  ├── api-endpoints.md (400줄)
  ├── page-structure.md (200줄)
  ├── component-library.md (350줄)
  └── workflows.md (250줄)
```

**효과**: "md만 읽고 구조를 파악" 가능

#### 전략 2: PR 가이드 체크리스트
```markdown
## PR 체크리스트

코드 작성 시:
- [ ] 타입 안정성 확인
- [ ] 에러 핸들링 추가
- [ ] 테스트 케이스 작성

문서 업데이트:
- [ ] API 변경 시 api-endpoints.md 업데이트
- [ ] 새 컴포넌트 추가 시 component-library.md 업데이트
- [ ] DB 스키마 변경 시 db-schema.md 업데이트
```

**효과**: "PR 타이밍에 항상 문서가 업데이트됩니다"

#### 전략 3: 레이어드 문서 구조
```
Layer 1: 핵심 가이드 (200줄)
  ↓ 필요 시 참조
Layer 2: 도메인 문서 (각 300줄)
  ↓ 필요 시 참조
Layer 3: 상세 스펙 (각 500줄)
```

**규칙**: AI는 항상 Layer 1부터 읽기 시작

#### 전략 4: 코드 > 주석
```typescript
// ❌ 나쁜 예: 장황한 주석
// 이 함수는 사용자 인증을 처리합니다.
// 먼저 토큰을 검증하고, 유효하지 않으면 에러를 던집니다.
// 유효하면 사용자 정보를 반환합니다.
function authenticateUser(token: string) {
  // ...
```

## ⚠️ 주의사항

### 1. AI는 만능이 아닙니다
```
AI가 잘하는 것:
✅ 반복적인 코드 작성
✅ 패턴 따라하기
✅ 테스트 케이스 생성
✅ 문서 정리

AI가 못하는 것:
❌ 비즈니스 의사결정
❌ 창의적인 UX 설계
❌ 복잡한 아키텍처 결정
❌ 보안 취약점 완벽 탐지
```

**인간의 역할**: 의도, 의사결정, 최종 검증

### 2. 컨텍스트 과부하 경계
```
증상:
- AI 응답이 느려짐
- 엉뚱한 코드 생성
- 토큰 비용 급증

예방:
- CLAUDE.md 200줄 엄수
- 도메인별 문서 분리
- 휘발성 컨텍스트 적극 활용
```

### 3. 보안은 인간이 책임
```
AI 코드 생성 후 필수 체크:
- [ ] XSS 취약점 확인
- [ ] SQL Injection 방지
- [ ] 민감 정보 노출 점검
- [ ] 인증/인가 로직 검증
```

### 4. 의존성 함정 주의
```
문제: AI가 추천한 라이브러리가 deprecated

해결:
1. package.json의 버전 명시
2. 주요 라이브러리는 CLAUDE.md에 명시
3. 정기적인 의존성 업데이트
```

### 5. 과도한 자동화 지양
```
❌ 모든 걸 AI에게 맡기기
✅ AI는 "고급 알바", 인간은 "감독자"

균형점:
- 반복 작업: AI
- 핵심 로직: 인간 설계 → AI 구현 → 인간 검증
```

```markdown
  "total_score": 24,
  "recommendation": "approve|reject|review",
  "reasoning": "간결한 한 문장 설명"
}

기준: total_score 24점 이상 approve, 18점 이하 reject, 나머지 review
```
